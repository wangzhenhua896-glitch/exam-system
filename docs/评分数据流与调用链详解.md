# 评分数据流与调用链详解

> 最后更新: 2026-04-15
> 本文档说明评分引擎的数据来源、调用链路、各科目差异，以及容易混淆的数据存储设计。

---

## 1. 一句话总结

**所有科目的评分数据来源是 `question_answers` 表，不是 `questions` 表。**

`questions` 表中的 `rubric_points`、`standard_answer`、`rubric_script` 等字段是**编辑器展示用**的中间产物，评分引擎**不直接读取**。

---

## 2. 调用入口

评分有两个入口，都最终调用同一个函数：

| 入口 | 路由 | 说明 |
|------|------|------|
| 单题评分 | `POST /api/grade` | `routes_grading.py:57` → `grade_answer()` |
| 批量评分 | `POST /api/batch` | `routes_grading.py:602` → `create_batch()` |

两处都调用 `three_layer_grade()` (见 `three_layer_grader.py:124`)。

**还有一个绕过三层评分的入口（遗留问题）：**
- `POST /api/verify-rubric`（测试用例验证）→ `routes_rubric.py:441` → 直接调用 `grading_engine.grade()`，**跳过了关键词匹配和向量匹配**。

---

## 3. 完整调用链

```
POST /api/grade
  │
  ├─ 1. 加载题目数据（如有 question_id）
  │     ├─ get_question(id) → 题目信息
  │     ├─ get_question_answers(id) → 满分答案 + 采分点
  │     └─ 英语父题: get_child_questions(id) + 每个子题的 get_question_answers()
  │
  ├─ 2. 空答案检查 → 返回 0 分
  │     ├─ 英语: is_empty_answer_en() (按词数 < 2 判断)
  │     └─ 其他: 去空格标点后长度 < 2
  │
  ├─ 3. 敏感词扫描 → 高危词命中返回 0 分
  │
  └─ 4. three_layer_grade()
        │
        ├─ 英语? ──YES──→ grade_english() [独立流程，见第4节]
        │
        └─ 其他科目 → 三层并行 [见第5节]
              ├─ 第1层: keyword_match_score() [同步]
              ├─ 第2层: vector_match_score() [线程池]
              └─ 第3层: grading_engine.grade() [线程池]
                    └─ 按 subject 选不同提示词
```

---

## 4. 英语科目评分流程（独立，不走三层并行）

英语在 `three_layer_grade()` 中直接跳出，交给 `grade_english()` (`english_grader.py:591`)。

### 4.1 数据来源

评分引擎从 `question_answers` 表中读取两类数据：

| scope_type | 存储内容 | 读取者 |
|-----------|---------|--------|
| `question` | 父题的满分答案文本 | `grade_english_fallback()` 构造 prompt 时引用 |
| `scoring_point` | **子题的采分点 JSON** | **主评分逻辑的核心输入** |

**关键：采分点 JSON 在子题上，不在父题上。**

例如 Q171 "Chinese Tea Culture" 是父题，它的 3 个子题是 Q193/Q194/Q195：
- Q193 的 `question_answers` 行中 `scope_type='scoring_point'` 存着 `{"id":"Q1", "scoring_points":[...]}`
- 这就是 `english_scoring_point_match()` 直接消费的输入

### 4.2 逐子题评分

```
for each scope_type='scoring_point' 的行:
    sp_json = json.loads(row['answer_text'])
    │
    ├─ english_scoring_point_match(answer, sp_json, sub_max)
    │   ├─ 预检: 中文字符 → 0分; 空答案 → 0分; 拼音 → 0分
    │   ├─ 排除词检查: 命中排除词 → 0分
    │   ├─ 逐采分点匹配关键词/同义词
    │   └─ 两种计分公式:
    │       ├─ max_hit_score: 取命中中最高分
    │       └─ hit_count: 按命中数查 rules 表
    │
    ├─ 如果精确匹配 = 满分 → 直接得分，跳过 LLM
    └─ 如果未满分 → grade_english_fallback() (LLM 兜底)
        ├─ 用采分点 JSON 构造 prompt
        ├─ LLM 判断语义等价
        └─ 取 max(精确匹配分, LLM分) → 对学生有利
```

### 4.3 最终得分

各子题最优得分之和，策略固定为 `max`。

### 4.4 无采分点回退

如果没有任何 `scope_type='scoring_point'` 的行（如未配置采分点的旧题），直接调用通用 LLM 评分 `grading_engine.grade()`，退化为和其他科目一样的流程。

---

## 5. 非英语科目评分流程（三层并行）

思政 (politics)、语文 (chinese)、通用 (general) 都走三层并行。

### 5.1 数据来源

同样从 `question_answers` 表读取两类数据：

| scope_type | 存储内容 | 第几层读取 |
|-----------|---------|-----------|
| `question` | 标准答案文本（满分答案） | 第2层（向量匹配） |
| `scoring_point` | 采分点关键词文本 | 第1层（关键词匹配） |

### 5.2 三层执行

```
第1层 keyword_match_score() [同步，快速]
  ├─ 筛选 scope_type='scoring_point' 的行
  ├─ 去空格后做子串匹配
  └─ 得分 = max_score / 采分点数 × score_ratio（命中项之和）

第2层 vector_match_score() [ThreadPoolExecutor]
  ├─ 筛选 scope_type='question' 的行（满分答案）
  ├─ text2vec 计算相似度
  └─ 得分 = 最高相似度 × max_score

第3层 grading_engine.grade() [ThreadPoolExecutor]
  ├─ 按 subject 选提示词:
  │   ├─ politics → "严格按照评分脚本" + "政治立场错误判0分"
  │   ├─ chinese  → "引用诗句后正常给分" + "错别字每处0.5分" + 支持负分
  │   └─ general  → 通用提示词
  ├─ rubric 中的 rubricScript 优先作为评分依据
  └─ 输出 scoring_items JSON
```

### 5.3 最终得分

```
final_score = _apply_strategy(layer_scores, strategy)
```

| strategy | 行为 |
|----------|------|
| `avg` | 三层均值（全局默认） |
| `max` | 取最大值 |
| `min` | 取最小值 |
| `median` | 取中位数 |

None 值（某层失败/跳过）被过滤，不参与计算。

策略优先级：`questions.scoring_strategy`（单题覆盖） > `grading_params` 表全局默认。

### 5.4 语义校验（非英语独有）

LLM 返回 `scoring_items` 后，会经过 `validate_scoring_items()` 二次校验，用 text2vec 检查是否有被漏判的等价表述，自动纠偏。**英语不走此步。**

## 6. 各科目差异对比

### 6.1 总览

| 维度 | politics | chinese | general | english |
|------|----------|---------|---------|---------|
| 评分路径 | 三层并行 | 三层并行 | 三层并行 | 独立流程 `grade_english()` |
| 第1层（关键词） | 从 `question_answers` 取 `scoring_point` 子串匹配 | 同左 | 同左 | 用 `scoring_point` JSON 做精确匹配（词边界+同义词） |
| 第2层（向量） | text2vec 相似度 | 同左 | 同左 | **跳过**（中文 text2vec 对英文不可靠） |
| 第3层（LLM） | 政治专属提示词 | 语文专属提示词 | 通用提示词 | `grade_english_fallback()`（采分点构造 prompt） |
| 策略取分 | avg/max/min/median | 同左 | 同左 | 固定 max（逐子题取最优） |
| 语义校验 | 有 | 有 | 有 | **无** |
| 子题处理 | 无子题概念 | 无子题概念 | 无子题概念 | 父题下挂子题，逐子题独立评分 |
| 采分点来源 | `question_answers` (`scope_type='scoring_point'`, 纯文本关键词) | 同左 | 同左 | `question_answers` (`scope_type='scoring_point'`, 结构化 JSON) |

### 6.2 LLM 提示词差异（qwen_engine.py）

三个非英语科目的 LLM 评分共享同一套框架，但提示词不同（`qwen_engine.py:154-162`）：

**通用提示词** (`_get_system_prompt`, 行 307-337):
- 要求逐项评分，输出 `scoring_items` JSON
- 总分由系统累加，模型不负责算总分
- 无科目特殊规则

**思政提示词** (`_get_politics_system_prompt`, 行 339-360):
- 强调"严格按照评分脚本执行"
- 政治立场错误判 0 分
- 等价表述按脚本中的表匹配
- 逐个要点说明判断结果

**语文提示词** (`_get_chinese_system_prompt`, 行 362-392):
- 引用诗句后有补充分析应正常给分
- 错别字按扣分规则执行（每处 0.5 分）
- **支持负分 scoring_item**（错别字扣分项的 score 为负值）
- 其他科目不支持负分

**英语提示词** (单独体系，`app/english_prompts.py`):
- 由 `grade_english_fallback()` 直接构造，不走 `_get_*_system_prompt` 的框架
- prompt 中嵌入了具体的采分点 JSON（keywords + synonyms + exclude_words）
- 让模型判断学生答案是否语义等价于某个采分点

### 6.3 Rubric 格式化差异（_format_rubric）

非英语科目：`qwen_engine.py:417-473`

| 优先级 | 数据来源 | 说明 |
|--------|---------|------|
| 1 | `rubricScript`（评分脚本） | 保证确定性，最优先 |
| 2 | `rubricRules` + `rubricPoints` | 规则+要点格式化展示 |
| 3 | `knowledge` / `contentType` / `aiPrompt` / `standardAnswer` | 辅助信息 |

英语科目：调用 `format_rubric_en()` 单独处理。

### 6.4 语义校验（非英语独有）

LLM 返回 `scoring_items` 后，非英语科目会经过 `validate_scoring_items()` 二次校验（`qwen_engine.py:250-277`）：
- 用 text2vec 计算学生答案与满分答案的向量相似度
- 检查是否有被 LLM 漏判的等价表述
- 发现漏判则自动纠偏，提高得分

**英语跳过此步**，因为中文 text2vec 模型对英文不可靠。

### 6.5 英语独立评分的内部差异

`grade_english()` 内部也有分支（`english_grader.py:591-811`）：

**有采分点** → 逐子题精确匹配：
- 两种计分公式（`score_formula` 字段）：
  - `max_hit_score`（`english_grader.py:209-264`）：遍历所有采分点，取命中中最高分的那个
  - `hit_count`（`english_grader.py:266-321`）：统计命中数，按 `rules` 查分表
- 精确匹配满分 → 直接得分，跳过 LLM（快速路径）
- 精确匹配未满分 → `grade_english_fallback()` LLM 兜底
- 取 `max(精确匹配分, LLM分)` → 对学生有利

**无采分点** → 退化为通用 LLM 评分：
- 直接调用 `grading_engine.grade()`，和其他科目一样
- 策略用 `max`（`english_grader.py:649`）
- 日志标记为"无采分点数据，回退通用评分"

### 6.6 非英语科目内部的得分层数波动

同一个思政/语文/通用题目，实际执行的评分会根据数据配置而变化：

| 情况 | 第1层 | 第2层 | 第3层 | 最终得分 |
|------|-------|-------|-------|---------|
| 有 scoring_point + 有满分答案 | 关键词匹配分 | 向量相似度分 | LLM 分 | 三层按策略取分 |
| 有 scoring_point + 无满分答案 | 关键词匹配分 | None（跳过） | LLM 分 | 两层按策略取分 |
| 无 scoring_point + 有满分答案 | None（跳过） | 向量相似度分 | LLM 分 | 两层按策略取分 |
| 无 scoring_point + 无满分答案 | None（跳过） | None（跳过） | LLM 分 | 只有 LLM 单层 |

None 值被 `_apply_strategy()` 过滤，不参与策略计算。所以同一科目内的不同题目，可能走三层、两层、或单层评分。

## 7. questions 表 vs question_answers 表：数据存储职责

### questions 表 — 编辑器展示 + 题目元数据

| 字段 | 内容 | 谁读 |
|------|------|------|
| `rubric` | 编辑器保存的完整 rubric JSON（含 type/version） | 编辑器回显 |
| `rubric_points` | **纯文本**，每行一个要点+分值，如 `green tea (2分)` | **仅前端卡片展示** |
| `rubric_script` | 评分脚本（非英语科目用） | 格式化后传给 LLM |
| `standard_answer` | 标准答案文本 | 编辑器展示；旧版迁移到 question_answers |
| `rubric_rules` | 评分规则文字描述 | 格式化后传给 LLM |

### question_answers 表 — 评分引擎的实际数据源

| scope_type | 存储内容 | 评分引擎如何用 |
|-----------|---------|---------------|
| `question` | 满分答案文本 | 第2层向量匹配的标准答案；LLM prompt 引用 |
| `scoring_point` | **非英语**：采分点关键词文本 | 第1层关键词匹配的匹配对象 |
| `scoring_point` | **英语**：结构化 JSON（含 `scoring_points` 数组） | `english_scoring_point_match()` 的核心输入 |

### 英语采分点 JSON 结构

存放在子题的 `question_answers` 行（`scope_type='scoring_point'`）中：

```json
{
  "id": "Q1",
  "text": "What is Longjing tea?",
  "max_score": 2,
  "score_formula": "max_hit_score",
  "scoring_points": [
    {
      "id": "1",
      "score": 2,
      "keywords": ["green tea"],
      "synonyms": ["a type of green tea"],
      "exclude_words": []
    }
  ]
}
```

### 为什么不直接用 questions.rubric_points？

1. `rubric_points` 是纯文本，不是 JSON，评分引擎无法解析
2. 一个父题挂多个子题，每个子题有独立的采分点，rubric_points 没有子题分区
3. `question_answers` 表天然支持子题粒度（每个子题一行），rubric_points 不行
4. `question_answers` 还同时存储满分答案（`scope_type='question'`），一个表搞定所有评分数据

## 8. 数据加载流程（评分前）

```
POST /api/grade (question_id=171)
  │
  ├─ get_question(171)
  │   └─ 拿到 subject, content, max_score, rubric, rubric_script, ...
  │
  ├─ 组装 rubric 字典（注入 rubric_script + rubric_rules + rubric_points + standard_answer）
  │   → 传给 LLM 引擎格式化为 prompt（非英语）或英语编辑器展示（英语）
  │
  ├─ get_question_answers(171)  → 父题的满分答案（scope_type='question'）
  │
  └─ 如果是英语父题:
       ├─ get_child_questions(171) → [Q193, Q194, Q195]
       ├─ get_question_answers(193) → Q193 的满分答案 + 采分点 JSON
       ├─ get_question_answers(194) → Q194 的满分答案 + 采分点 JSON
       └─ get_question_answers(195) → Q195 的满分答案 + 采分点 JSON
```

这些数据合并到 `question_answers_list` 中，传入 `three_layer_grade()`。

## 9. 容易踩的坑

### 9.1 写测试时不要从 questions.rubric_points 取采分点

```python
# 错误: rubric_points 是纯文本
row = db.execute("SELECT rubric_points FROM questions WHERE id=171").fetchone()
sp_json = json.loads(row['rubric_points'])  # JSONDecodeError!

# 正确: 从 question_answers 取
row = db.execute("""
    SELECT answer_text FROM question_answers
    WHERE question_id=193 AND scope_type='scoring_point'
""").fetchone()
sp_json = json.loads(row['answer_text'])  # 正确的 JSON
```

### 9.2 英语采分点在子题上，不在父题上

父题（如 Q171）没有 `scoring_point` 行。采分点 JSON 分散在它的子题（Q193/Q194/Q195）上。

### 9.3 非英语的 scoring_point 行是纯文本，英语是 JSON

非英语科目中 `scope_type='scoring_point'` 的 `answer_text` 就是一个关键词字符串（如 `"尊老爱幼"`），用于子串匹配。
英语科目中同样字段存的是完整 JSON，用于结构化匹配。

### 9.4 测试用例验证绕过三层评分

`POST /api/verify-rubric` 直接调用 `grading_engine.grade()`，不走关键词匹配和向量匹配。这意味着验证通过不等于实际评分时三层都通过。
