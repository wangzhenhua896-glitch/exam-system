# 评分脚本自查自纠方案

## 目标

验证评分脚本的**正确性**和**完整性**，确保评分引擎能准确、一致地打分。

## 四层检查体系

### 第一层：结构检查（自动化）

代码层面检查，无需人工介入。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 8个必选章节 | 正则匹配章节标题 | 题目信息、标准答案要点、逐项评分规则、反作弊规则、作答情况分类、扣分规则、总分计算、输出格式 全部存在 |
| JSON 输出格式 | 提取输出格式中的 JSON 模板，`json.loads()` 验证 | 解析成功，无语法错误 |
| scoring_items 分值 | 从 JSON 模板提取 `full_score`，求和 | 总和 == max_score |
| 满分标注 | 搜索"满分N分" | 与 max_score 一致 |
| 要点数量 | 正则匹配"要点N：" | 数量 >= 1 |
| 无模糊用语 | 搜索"酌情"、"视情况"、"适当给分"、"根据质量" | 不应出现 |

### 第二层：内容对齐（自动化+人工）

与 Word 文档原始标准答案对比。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 要点数量 | Word原文要点数 vs 脚本要点数 | 一致 |
| 要点名称 | Word原文要点名 vs 脚本要点名 | 核心含义一致（允许表述差异） |
| 各要点分值 | Word原文分值 vs 脚本分值 | 每个要点分值一致 |
| 分值合计 | 各要点分值之和 | == max_score |
| 等价表述覆盖 | 人工审核 | 每个要点至少有 3 个等价表述 |
| 评分规则 | Word原文评分规则 vs 脚本规则 | 逻辑一致（关键词判断、部分得分等） |

### 第三层：评分验证（跑测试用例）

用 126 个测试用例调用评分 API，对比实际得分与期望分数。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 得分准确性 | `actual_score vs expected_score` | 误差 <= 1.0 分 |
| 通过率 | 通过用例数 / 总用例数 | >= 90% |
| 零分用例 | 空白/放弃/答非所问的得分 | 必须为 0 分 |
| 满分用例 | 完整答案的得分 | 必须为满分 |
| 场景覆盖 | 按场景类型统计通过率 | 每个场景类型通过率 >= 80% |

执行方式：
```bash
# 调用 /api/grade 接口，对每个测试用例评分
# 将 actual_score 写回 test_cases.last_actual_score
# 将 error = |actual - expected| 写回 test_cases.last_error
```

### 第四层：一致性验证（重复跑）

同一答案多次评分，验证分数是否一致。

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 单题一致性 | 同一答案连续评 3 次 | 3 次分数完全相同 |
| 批量一致性 | 全部 126 个用例各评 3 次 | 每个用例 3 次分数一致 |
| 评语区分度 | 相同分数的不同答案 | 评语应有差异，不应完全相同 |

执行方式：
```bash
# 每个用例调用 /api/grade 3 次
# 对比 3 次的 score 是否完全一致
# 记录不一致的用例
```

## 实施步骤

1. **第一层**：写脚本批量检查全部 12 道思政题的评分脚本结构
2. **第二层**：逐题对比 Word 文档和数据库中的要点、分值
3. **第三层**：跑 126 个测试用例，生成评分结果报告
4. **第四层**：抽样跑一致性（选 10 个典型用例各评 3 次）

## 输出

- 自查报告：每层检查结果，通过/未通过，问题清单
- 修复清单：需要修改的评分脚本及具体修改内容
- 修复后复验：修改后重新跑全部四层检查
