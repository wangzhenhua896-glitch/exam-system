# 数据库版本管理说明

> 更新日期：2026-04-15

---

## 零、Gitea 与 GitHub 的根本区别

**Gitea 是本地私有仓库**（运行在局域网 192.168.255.10），数据不出局域网，因此：

- ✅ 可以放真实题库数据、评分记录、学生数据
- ✅ 可以把数据库文件直接提交进去
- ✅ API Key、配置等敏感信息理论上也可以放（但仍建议用 .env 分离）

**GitHub 是公开的云端仓库**，绝对不能放：

- ❌ 数据库文件（含学生答案、评分记录）
- ❌ API Key（会被机器人扫描盗用）
- ❌ 任何真实用户数据

> **本项目只用 Gitea，不用 GitHub。** 所以数据库提交进 Gitea 完全没问题。

## 一、当前存在几个数据库版本

| 位置 | 路径 | 说明 | 谁维护 |
|------|------|------|--------|
| 本地开发机 | `data/exam_system.db` | 开发/测试用，含大量测试记录 | 各 Agent / 王老师 |
| Gitea 仓库 | master 分支同名文件 | 随代码一起提交，保存基准题库 | git push 同步 |
| 阿里云服务器 | `/opt/ai-grading/data/exam_system.db` | 正式运行库，评分记录最多 | 独立维护，部署时不覆盖 |

> `deploy.sh` 已排除 `data/` 目录，**部署不会覆盖服务器数据库**。

---

## 二、为什么数据库被纳入 git（与 .gitignore 矛盾）

`.gitignore` 里写了 `data/*.db`，按理不应追踪。  
但早期某次提交（`9a2a416`）把数据库直接 `git add` 进去了，**已追踪的文件不受 .gitignore 保护**，之后每次都带着一起提交了。

当前状态：**数据库是被追踪的**，每次 `git push` 都会把最新的库推到 Gitea。

---

## 三、这样管理的利弊

### 优点
- Gitea 上随时有一份可用的基准库，新机器 `git clone` 直接能跑
- 题目、采分点、模型配置等"主数据"随代码一起版本化

### 缺点
- 数据库是二进制文件，git 看不出"改了什么"，只知道"变了"
- 评分记录（`grading_records`）、测试用例等运行数据也跟着进了 git，不够干净
- 阿里云的真实评分数据永远不会同步回来

---

## 四、推荐的日常操作方式

### 情况 A：只改了代码，没动数据库
正常 `git add` 代码文件，**不要** `git add data/exam_system.db`，避免把无关的运行数据带进提交。

### 情况 B：更新了题库/采分点（主数据）
如本次补全英语题采分点，应该：
1. 把变更逻辑写进 `scripts/` 下的脚本（已做）
2. **同时提交数据库**，让 Gitea 上的库保持最新可用状态

```bash
git add scripts/import_english_scoring_points.py data/exam_system.db
git commit -m "feat: 补全英语采分点数据"
git push
```

### 情况 C：想彻底把数据库从 git 中移除
```bash
# 从追踪列表移除（不删除本地文件）
git rm --cached data/exam_system.db
# 然后 .gitignore 的规则才会生效
git commit -m "chore: 将数据库移出版本控制"
```
移除后 Gitea 上就没有数据库了，新机器需要用脚本重建（见下方）。

---

## 五、如果需要在新机器重建数据库

数据库结构和主数据都可以从脚本重建：

```bash
# 1. 建表结构（db_models.py 启动时自动建表）
python main.py &
# Ctrl+C 停掉，表结构已建好

# 2. 导入英语题采分点（0001-0012）
python scripts/import_english_scoring_points.py

# 3. 其他批量导入脚本（如有）
python scripts/import_english_questions.py
```

> 题目正文（原文、评分脚本）存在 `questions` 表，目前只在 Gitea 数据库中，
> 尚未单独导出为 SQL 或 JSON。如需迁移，可用：
> ```bash
> sqlite3 data/exam_system.db ".dump questions" > questions_dump.sql
> ```

## 六、三个环境的数据同步原则

```
本地开发机  ──git push──▶  Gitea（基准库）
                              │
                       手动/deploy.sh
                              │
                              ▼
                         阿里云服务器（不覆盖，独立运行）
```

- **本地 → Gitea**：改了主数据就推，改了运行数据不推
- **Gitea → 阿里云**：`deploy.sh` 不同步 `data/`，需手动按需操作
- **阿里云 → 本地**：如需拿回真实评分数据，手动 `scp` 或 `sqlite3 .dump`
