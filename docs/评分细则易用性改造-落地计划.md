# 评分细则易用性改造 — 落地计划

## 背景

"评分脚本"（rubric_script）是引擎的**唯一依据**（LLM 直接读取原始文本打分），但老师完全看不懂它。
系统已经知道这个问题（"评分脚本无模板引导"曾被标记为 P2），但一直没有系统性解决。

术语统一：**UI 展示全部改为"评分细则"**，代码内部变量名暂不变。

**目标**：老师不需要理解脚本格式，就能配置、审核、理解评分规则。

---

## 三条路线（按依赖关系排序）

```
路线2（结构化卡片）→ 路线1（表单编辑）→ 路线3（自然语言描述）
```

---

### 路线2：评分细则 → 结构化卡片展示（先做）

**目标**：把原始脚本文本解析成老师能看懂的卡片，替代 monospace textarea。

**核心思路**：
- 解析 8 个 `【...】` 段落，提取得分点、关键词、分数、反作弊规则
- 用 `el-collapse` 卡片展示，每段可展开/收起
- 关键词显示为 `el-tag` 芯片，分数用 badge
- 保留"原始文本"切换按钮作 fallback
- **不改数据，不改引擎，纯展示层**

#### MVP：英语编辑器第3步改造

英语编辑器第3步（"评分脚本"）是最自然的起点，因为：
- 老师最常用此入口
- `subQuestions` 数组已包含完整结构化数据（scoringPoints、keywords、synonyms）
- **不需要解析器**——数据已经是结构化的，只需替换 textarea 为卡片渲染

**改动文件**：
| 文件 | 操作 | 说明 |
|------|------|------|
| `templates/question-bank.html` | 修改 | 第3步：textarea → 卡片展示 subQuestions 的评分细则 |
| `static/js/englishEditCore.js` | 修改 | 步骤名"评分脚本"→"评分细则"，加视图切换状态 |

**第3步新 UI 设计**（替换 lines 533-537 的 textarea）：

```
[ 评分细则视图 ] [ 原始脚本 ]    ← 切换按钮，默认显示细则视图

── 评分细则视图 ──
┌─ 第1题（2分）─────────────────────┐
│  题目：What is the capital of...  │
│  计分公式：max_hit_score          │
│                                  │
│  采分点 A（1分）                   │
│  ├─ 关键词：London, capital       │
│  └─ 等价表述：The capital city... │
│                                  │
│  采分点 B（1分）                   │
│  ├─ 关键词：United Kingdom, UK    │
│  └─ 等价表述：Britain, England... │
│                                  │
│  排除词：Paris, France            │
└──────────────────────────────────┘
┌─ 第2题（4分）─────────────────────┐
│  ...                              │
└──────────────────────────────────┘
│ 共 3 小问，满分 8 分               │
```

- 每个子题用 `el-collapse-item` 展示
- 每个采分点：关键词用蓝色 `el-tag`，等价表述用绿色 `el-tag`（outline）
- 排除词用红色 `el-tag`
- hit_count 公式的规则用表格展示
- "原始脚本"切换显示现有 textarea（保留自查/质量评估功能）

**不需要新建 `rubricParser.js`**——MVP 阶段直接用 `subQuestions` 渲染，零解析成本。

**验证标准**：
- 老师进入英语编辑器 → 第2步配置完采分点 → 进入第3步看到的是结构化卡片而非脚本文本
- "原始脚本"切换可查看/编辑原始文本
- 自查/质量评估/生成脚本功能不受影响
- 保存后重新加载，卡片正确展示

### 路线1：卡片 → 表单编辑（路线2 完成后做）

**目标**：把只读卡片变成可编辑表单，老师通过表单配置评分规则，系统自动生成脚本。

**核心思路**：
- 新增 `generateScript(data)` 函数：结构化数据 → 脚本文本（解析器的逆操作）
- 得分点：`el-input` 输入名称、`el-input-number` 输入分值、tag 芯片输入关键词/等价表述
- 反作弊规则：可增删的列表，每条 `el-input`
- 生成的脚本文本继续存入 `rubric_script` 列，引擎照常消费
- 保存前做 round-trip 验证：`generateScript(parseScript(text))` 与原文等价

**英语编辑器**：路线1 第一阶段先改造英语编辑器第2步的"逐小问配置"——让老师直接在卡片上编辑，跳过第3步（脚本自动生成+保存在后台完成，老师不再看到脚本）。

**思政/语文/通用**：把英语编辑器的表单模式推广到其他科目（需要 rubricParser.js 提供 parse + generate 的双向能力）。

**改动文件**：
| 文件 | 操作 | 说明 |
|------|------|------|
| `static/js/rubricParser.js` | 修改 | 新增 `generateScript()` 逆向生成函数 |
| `static/js/rubricCardEditor.js` | **新建** | 可编辑卡片组件、表单状态管理 |
| `dist/index.html` | 修改 | 编辑模式用表单卡片替代 textarea |
| `templates/question-edit.html` | 修改 | 脚本编辑区改为表单 |
| `templates/rubric-workbench.html` | 修改 | 脚本编辑区改为表单，保留 AI 工具按钮 |
| `static/js/useRubric.js` | 修改 | round-trip 验证、表单-脚本同步 |
| `templates/question-bank.html` | 修改 | 英语编辑器第2/3步合并，表单即细则 |

**难度**：L（round-trip 保真性是最大挑战）

**关键风险**：
- `generateScript(parseScript(text))` 可能丢失格式细节 → 需要对所有现有脚本做 round-trip 测试
- 生成的文本必须符合引擎期望的模式（引擎传 verbatim 给 LLM）

## MVP 技术风险分析（已完成）

### 数据恢复可靠性：通过

`loadChildren()`（englishEditCore.js 293-376行）从 DB 恢复 subQuestions 的完整流程：
- 子题 content / standard_answer / max_score → 直接从子题记录取
- rubric JSON → 提取 scoreFormulaType + scoringRules
- question_answers（scope_type=scoring_point）→ JSON.parse(answer_text) → scoringPoints + keywords + synonyms + excludeList + pinyinWhitelist

**结论**：进入第3步时 subQuestions 数据基本完整。唯一边缘情况是老数据缺少 scoring_point 记录（scoringPoints 为空），此时卡片显示"无采分点配置"并 fallback 到原始脚本。

### 5 个风险点

| # | 风险 | 严重程度 | MVP 对策 |
|---|------|---------|---------|
| 1 | subQuestions 为空时卡片无法渲染 | 中 | existing workflow gating 保证进入第3步时必有子题；空采分点时 fallback 到原始脚本 |
| 2 | 脚本生成变后台，自查结果如何呈现 | 中 | MVP 保留"原始脚本"切换，自查/质量评估功能不受影响 |
| 3 | 第2/3步边界模糊 | 低 | 先做出来让老师体验，根据反馈迭代 |
| 4 | 卡片与脚本不一致（改了采分点未重新生成） | 中 | 进入第3步时检测 subQuestions 变化，自动重新生成或提示 |
| 5 | 保存逻辑 | 低 | 无风险，saveAll() 保存 generatedScript 文本，不受展示层改动影响 |
