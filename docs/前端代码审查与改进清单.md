## P0 — 必须修复（影响功能正确性或用户体验）

### P0-1：评分脚本编辑器滚轮劫持

- **文件**：question-edit.html（rubricScript watcher）
- **问题**：watcher 监听 `questionForm.rubricScript`，每次内容变化都执行 `scrollTop = 0`。用户编辑长脚本时每敲一个字页面跳回开头，无法正常使用。
- **根因**：watcher 没有区分"加载新脚本"和"用户编辑"两种场景。
- **建议**：只在内容从空变为非空（加载新题目脚本）时重置滚动位置，用户编辑过程中不触发。

### P0-2：删除操作吞掉所有错误

- **文件**：question-edit.html, dedup.html, admin.html（delete 操作 catch 块）
- **问题**：catch 块用 `/* cancelled */` 把所有异常（网络错误、服务器 500）全部静默吞掉。如果 `axios.delete` 真的失败，用户没有任何提示，以为删除成功但数据还在。
- **根因**：ElMessageBox.confirm 用户取消时抛字符串 `'cancel'`，网络/服务器错误抛 `Error` 对象，但 catch 块没有区分这两种情况。
- **建议**：用 `e instanceof Error` 区分，异常时弹出 `ElMessage.error`。

### P0-3：多处加载失败无用户提示

- **文件**：question-bank.html + question-edit.html（loadStats, loadDashboard, loadQuestions, loadTestCases, loadAnswers 等）
- **问题**：5 个数据加载函数失败时只打 `console.error`，用户看到空白页面/空列表，不知道是加载失败还是真的没数据。
- **建议**：在 `console.error` 之后加 `ElMessage.error('加载XXX失败：' + 错误原因)`，让用户知道出了什么问题。

### P0-4：JSON.parse 无保护导致白屏

- **文件**：question-bank.html（currentSubjectLabel computed）
- **问题**：`currentSubjectLabel` computed 属性直接 `JSON.parse(localStorage.getItem(...))`，如果 localStorage 被污染（非 JSON 格式），整个页面白屏。
- **建议**：加 try-catch，解析失败时降级返回原始科目标识。

### P0-5：英语编辑器阅读材料丢失格式

- **文件**：question-bank.html + englishEditCore.js（Step1 原题解析）
- **问题**：英语编辑器 Step1（原题解析）中，阅读材料用 `<el-input type="textarea">` 显示。后端 `original_text` 存的是完整 HTML（`<p>`、`<span style="font-size:14.0pt">` 等），但 textarea 中全部降级为纯文本，格式丢失。
- **根因**：`englishEditCore.js` 加载时只取 `parent.content`（纯文本），通过 `extractReadingMaterial()` 剥掉所有 HTML 标签后存入 `parentContent`，模板中用 textarea 显示。
- **建议**：新增 `parentContentHtml` 状态存储 HTML 版本（来自 `original_text` 或 `content_html`），默认用 `v-html` 渲染，提供"编辑文本"按钮切换到 textarea 模式。

## 架构层面问题

### 两套模块系统并存

当前前端同时存在两种模块模式：
1. **IIFE 全局变量**（window.SharedApp, window.EnglishEditCore）：shared.js、englishEditCore.js 和所有模板页面的行内 script
2. **ES Modules**（import/export）：dist/index.html 使用的 use*.js 模块

两套系统无法互相引用。ES Module 中无法直接用 `window.SharedApp`（虽然技术上可以但不规范），IIFE 中无法用 import。长期来看应该统一。

### 5 个模板页面没有使用 composable 架构

login.html、admin.html、test-cases.html、sensitive-words.html、user-management.html、question-bank.html、question-edit.html、dedup.html、import.html、ai-generate.html、syllabus.html 的 Vue 逻辑都是行内 `<script>`，使用 IIFE 全局变量模式。随着功能增长，这些页面也会面临巨型单文件问题。

### dist/index.html 和 question-bank.html 大量重复

dist/index.html 是评分功能的独立页面（~856 行），与 question-edit.html 的评分脚本生成/验证逻辑有重叠。应该确认 dist/index.html 的定位——如果是独立部署用，考虑通过构建工具自动生成而不是手动同步。
