## 五、问题总结与优先级

### P0 - 严重

1. **GLM-4-7 不适合生产使用**
   - 平均响应 74 秒（其他模型 9-26 秒），超时率 25%
   - V7 近义表达测试 3 次全部超时
   - 建议：禁用或标记为"实验性"

2. **近义表达识别能力严重不一致**
   - 2 个模型能正确识别，3 个不稳定，2 个完全无法识别
   - 偏差最大达 2.33 分（满分 8 分），等同于一个要点全部错判
   - 建议：在评分脚本中增加"等价表达对照表"，明确告诉模型哪些词等价

### P1 - 重要

3. **同一模型多次评分不稳定**
   - Seed 2.0 Code 对 V7 评出了 4/6/7 三个不同分数（波动 3 分）
   - DeepSeek 系列对 V7 也有 2 分波动
   - 原因：temperature=0 理论上应消除随机性，但模型对近义表达的判断本身存在不确定性
   - 建议：对近义表达类答案启用多次采样取平均

4. **API 超时无自动重试**
   - 后端引擎有 3 次重试，但前端测试中直接失败
   - 建议：runner.py 增加重试逻辑；同时排查 GLM 超时的根因

### P2 - 优化

5. **响应耗时差异大**
   - 最快 9 秒 vs 最慢 74 秒（不含 GLM），8 倍差距
   - 对批量评分（1000 份）影响巨大：最快 2.5 小时 vs 最慢 20 小时
   - 建议：生产环境中选择响应最快的模型作为默认

6. **`/api/providers` 的 `active` 字段语义问题**（已修复）
   - 原逻辑：`active` 表示"当前评分引擎选中的模型"
   - 修复后：`active` 表示"用户是否启用"，新增 `selected` 字段表示"引擎当前选中"

## 六、建议的下一步行动

1. **扩大测试覆盖**：用更多题目（语文、英语等）做同样测试，验证结论是否普适
2. **优化评分脚本**：在 rubricScript 中增加"等价表达对照表"，减少模型对近义词的理解歧义
3. **引入多次采样**：对高风险答案（近义表达、临界分数）自动启用 3 次采样取平均
4. **禁用 GLM-4-7**：或联系服务商排查超时问题
5. **建立常驻监控**：每次评分自动记录，定期跑一致性测试，持续监控评分漂移
